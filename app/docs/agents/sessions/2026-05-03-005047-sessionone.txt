  ---
  Question 2 of N — Replay format and core↔frontend contract

  The Python core is going to emit something the web frontend consumes for both
  live inspection and replay scrubbing. Three formats:

  1. Per-tick state snapshots only. Core emits the full state every tick (say 20
   Hz). Renderer interpolates positions between ticks for smooth motion.
  Simplest contract; no separate event types. Animation triggers (attacks, hits,
   deaths) must be derived by diffing consecutive snapshots — workable but
  fragile (e.g., a unit that died and respawned in the same tile would be
  missed).
  2. Initial state + event log only. Core emits one starting snapshot, then a
  stream of typed events (deploy, move, attack_start, projectile_fired, damage,
  destroyed, spell_cast). Renderer reconstructs state by applying events.
  Compact, animation triggers are first-class, exact replay determinism. Cost:
  scrubbing backward requires replaying from start (or periodic checkpoints).
  3. Hybrid: initial state + per-tick delta + animation events. Core emits a
  starting snapshot, then per tick: a small delta (only what changed) + a list
  of events that occurred this tick. Renderer can both render state at any tick
  and trigger animations. Best of both, slightly more code in the core's
  serializer.

  My recommendation: Path 3 (hybrid), but build it in phases. Start the tracer
  bullet with Path 1 (full state per tick) because it's the simplest thing that
  lets you stand up a renderer. As soon as you want attack animations to look
  good, add an events: [...] array per tick — that's a backward-compatible
  extension. Skip deltas until file size or network bandwidth actually hurts;
  for TH6 with maybe 200 entities, full-state JSON at 20 Hz for a 90-second
  attack is ~few MB, which is fine on disk and fine in memory.

  Concretely the per-tick payload would look like:

  {
    "tick": 412,
    "time_ms": 20600,
    "state": { "buildings": [...], "troops": [...], "projectiles": [...],
  "spells": [...], "score": {...} },
    "events": [
      {"type": "attack_start", "src": "troop_17", "tgt": "bld_4"},

  600},

    ]
  }

  And the core also emits a replay.json containing {seed, base_layout,

  Question 4 of N — Determinism, tick model, and time

  The simulator can run on a spectrum from "fully deterministic, seeded,
  frame-exact reproducible" to "physics-style continuous time with stochastic
  mechanics." Three choices to make as a bundle:

  1. Determinism level
    - (a) Fully deterministic with explicit seed. Every stochastic call (target
  tie-breaks, troop AI choices, spell jitter if any) goes through a single
  seeded RNG. Same (base, deployment_plan, seed) always produces bit-identical
  state. Replays are guaranteed perfect.
    - (b) Deterministic-but-no-randomness. No RNG at all. All tie-breaks
  resolved by canonical ordering (e.g., target with lowest id). Same input →
  same output trivially.
    - (c) Stochastic. RNG used freely, replays best-effort.
  2. Tick rate
    - Real Clash runs at 20 Hz internally (≈50 ms per tick). For TH6 a max
  attack is 3 minutes = 3600 ticks.
    - Choose one fixed rate (e.g., 20 Hz) and everything (movement speed, attack
   cooldown, projectile travel, spell duration) is expressed in ticks.
  3. Time semantics in RL
    - Does the RL agent get to act every tick? Or only at deployment moments? In
   real Clash you make ~30 deploy decisions over 3 minutes, then watch.
    - Two natural framings: per-tick action (agent emits "do nothing" most
  ticks, deploy occasionally) or decision-point episodes (env steps only when
  the agent has a choice — agent emits "deploy troop X at position (r,c)" or
  "wait Δt"; sim fast-forwards between decisions).

  My recommendation: (1b) deterministic-no-randomness for v1, (2) 20 Hz fixed 
  tick rate, (3) decision-point action model.

  Reasoning:
  - No RNG by default is a hard rule that pays off for years. Tie-breaks
  resolved by (lowest_id, then lowest_distance, then lowest_row_col) are boring
  and reproducible. Real Clash has tiny stochastic elements (Witch skeleton
  spawn timing, etc.) but TH6 has essentially none — Giants, Barbarians,
  Archers, Wizards are deterministic. Add seeded RNG later only if a v2 mechanic
   actually requires it. This makes every bug exactly reproducible from
  (base.json, plan.json) — a debugging superpower for an RL project.
  - 20 Hz matches the source material and gives smooth animation without
  exploding training cost. At decision-point granularity it doesn't matter for
  training throughput anyway.
  - Decision-point action model is enormously better for learning. A per-tick
  agent has to learn that 99% of actions should be "no-op" — a sparse credit
  assignment nightmare. A decision-point agent has an action space like
  (troop_type, row, col) and outputs ~30 actions per episode. Wait-actions can
  be a special "deploy_at_time(t)" that pre-schedules, or a "hold X seconds"
  action. This collapses the horizon from 3600 to ~30, which is the difference
  between "trains in days" and "trains in months."

  The cost: decision-point framing means the env's step() advances simulation by
   a variable amount of wall-time (until the agent's next decision is required),
   and the renderer needs to know how to play back that variable-length gap
  smoothly. The hybrid replay format already handles this (the tick field is the
   source of truth, not a step counter).

  Do you accept (deterministic-no-RNG, 20 Hz, decision-point action model)? Or 
  do you want to keep per-tick actions on the table because you want the agent 
  to also learn micro-management like dropping a single Archer between two 
  defense layers at the perfect moment?

  Where the 20 Hz comes from

  20 ticks/sec is the figure most commonly cited in Clash community
  documentation and private-server projects (CoCBot, etc.) as the internal
  simulation rate. It's not officially published. Some sources cite 30 Hz, some
  say render is 60 fps but logic is decoupled. The honest stance for this 
  project: the exact number doesn't matter for the research question. We just 
  need a consistent, fixed rate that makes balance numbers easy to author and 
  animation easy to render. I'd actually argue for 10 Hz (100 ms/tick) for v1
  because it makes balance arithmetic cleaner — but 20 Hz matches the source
  material if you ever want to compare to in-game footage. Either works. Pick
  one and never change it.

  What a tick is

  A tick is the smallest atomic unit of simulation time. Nothing happens between
   ticks. The world is a frozen photograph at tick 412, then a frozen photograph
   at tick 413. The core's job each tick is to compute photograph N+1 from
  photograph N.

  All quantities in the simulator are expressed in tick-relative units:
  - Movement speed = tiles per tick (e.g., Giant = 0.012 tiles/tick at 20 Hz)
  - Attack cooldown = ticks between attacks (e.g., Cannon = 16 ticks)
  - HP regen, projectile travel, spell duration = all in ticks
  - Damage is per-hit (event-based), not per-tick — but a per-tick damage source
   like a healing spell would be expressed as HP/tick

  Tick rate is decoupled from render framerate. The web viewer can run at 60 fps
   and interpolate troop positions between tick N and N+1; tick rate only
  governs simulation granularity.

  How a tick is computed (the per-tick update loop)

  Each tick the core does a fixed-order pass over the world. Order matters for
  determinism:

  def step_tick(state):
      1. Resolve projectiles in flight.    # arrows land, deal damage events
      2. Apply scheduled deployments.       # any troops the agent queued for
  this tick spawn now
      3. Update troop targeting.            # if target dead/destroyed, pick new
   (canonical tie-break)
      4. Move troops one substep.           # advance each troop by its speed*dt
   along its path
      5. Troops attack if in range + ready. # emit damage event or spawn
  projectile
      6. Update defense targeting.
      7. Defenses attack if in range + ready.
      8. Apply damage events.               # subtract HP, mark destroyed, emit
  "destroyed" event
      9. Update score (% destruction, stars).
      10. tick += 1
      return new_state, events_this_tick

  Step 4 is where pathfinding lives. A common cheap approach: each troop has a
  cached path (recompute on wall-break or target-death), and movement is just
  "advance the path-cursor by speed*dt." Movement is continuous-position on a 
  discrete grid — troop coordinates are floats (2.7, 4.3), but the grid they
  navigate is integer-indexed.

  How it shows up in the RL contract

  Decision-point step() looks like:

  def step(action):
      # action is one of: ("deploy", troop_type, row, col) or ("wait", Δticks)
  or ("end")
      if action[0] == "deploy":
          state.scheduled_deployments.append(...)
      fast_forward_until_next_decision_point()  # runs many ticks of step_tick()
      return obs, reward_delta, done, info

  A single RL step() may advance 1 tick or 200 ticks of simulation depending on
  what the agent chose. The agent never sees ticks directly — it sees
  observations at decision points. The renderer sees every tick's
  snapshot/events from the replay, so visuals are smooth regardless of agent
  decision cadence.

  Recommendation refinement

  For v1 I'd actually lean toward 10 Hz over 20 Hz because:
  - Balance numbers stay round (Giant moves 1 tile per second = 0.1 tiles/tick)
  - Half the ticks per attack episode → smaller replay files, cheaper sim
  - Animation interpolation in the web viewer makes the lower rate invisible to
  humans
  - Nothing in TH6 needs sub-100ms timing precision

  If we ever want to model frame-perfect Clash mechanics (Hog Rider jump timing,
   etc.) we can rerate later — all balance data is in JSON, not hardcoded.

  Want to lock in 10 Hz for v1, or stick with 20 Hz to stay closer to the 
  source? Then I'll move to the next question.

  Question 5 of N — Grid/map model, walls, building footprints

  The spatial substrate has surprising design depth and gates how everything
  else (action space, pathfinding, obs tensor, replay) is shaped. Real Clash
  uses a 44×44 tile grid with a center-aligned playable area and a deploy
  border. Three decisions to bundle:

  1. Grid size and coordinate system
    - (a) 44×44 with [0,0] at top-left, integer tiles for buildings, float 
  positions for troops. Matches source. Some tiles are non-buildable (border,
  water).
    - (b) A smaller grid for v1 (say 22×22) — half the cells, simpler
  pathfinding, less obs tensor, faster training. You can rescale later.
  2. Building footprints
    - Buildings in Clash occupy 2×2, 3×3, or 4×4 contiguous tiles. Town Hall =
  4×4, Cannon = 3×3, Wall = 1×1, etc. Footprints must be respected for placement
   and collision.
    - This needs to be a first-class concept in the core (Building has
  footprint: (w, h) and occupies [(r, c), (r+1, c), …]).
  3. Wall semantics — the trickiest one
    - (a) Walls as 1×1 tile occupants. Each wall segment is a building with HP.
  Pathfinder sees occupied tile = blocked. Wall-breakers do extra damage to
  walls. Simple, reasonably accurate.
    - (b) Walls as edges between tiles. More accurate to source (walls sit on
  tile boundaries, troops walk through a tile but can't cross a wall edge).
  Pathfinder needs edge-aware A*. Significantly more code.
    - (c) Walls as 1×1 tiles but with an "is_wall" flag so wall-breakers,
  jumpers (later), and pathfinder can treat them specially without polymorphism.

  My recommendation: (1a) full 44×44, (2) first-class footprints, (3c) walls as 
  1×1 tile occupants with is_wall flag.

  Reasoning:
  - Full 44×44 because shrinking the grid changes the strategic shape of the
  game (compaction ratios, wall layouts, defense coverage zones) in ways that
  may quietly break what the agent learns. It also makes a future Cartographer
  harder — real screenshots are 44×44 and you'd have to resample. The cost is
  mostly in the obs tensor size, which is fine for TH6 (44×44 with ~10 channels
  = 19k floats — trivially cheap on a GPU). Pathfinding on 44×44 with cached
  paths and incremental replanning is also fine at decision-point cadence.
  There's no upside to 22×22 once you've sized the obs tensor properly.
  - First-class footprints are non-negotiable; without them you can't place
  buildings correctly or compute "is this tile blocked" for pathfinding. Cost is
   one helper that returns the occupied tile set given (origin, footprint).
  - Walls as 1×1 tile occupants with is_wall flag because (b) edge-walls is a
  known pathfinding tar-pit that adds weeks of work and gives the agent ~zero
  additional learnable signal at TH6. The "tile-walls" approximation is what
  nearly every Clash sim project uses, and at 10 Hz with continuous troop
  positions the visual difference is invisible. The is_wall flag is so
  wall-breakers, healing spells (which target ground troops anyway), and the
  pathfinder can each handle walls with a one-line check without inheritance
  gymnastics.

  The tradeoff being accepted: with tile-walls, a troop adjacent to a wall is
  "in" the tile next to the wall, not "touching" the wall — so wall-breakers
  blow up on the wall tile (replacing themselves with the wall's hp drop), which
   is slightly off from source. For v1, fine.

  Concrete data structures this implies:

  @dataclass
  class Building:
      id: int
      type: str           # "cannon", "town_hall", "wall", "archer_tower", ...
      origin: tuple[int, int]   # top-left (row, col)
      footprint: tuple[int, int]  # (h, w)
      hp: int
      max_hp: int
      is_wall: bool
      is_destroyed: bool
      # defense-only fields:
      target_id: int | None = None
      cooldown_remaining: int = 0   # in ticks

  Plus a derived tile_occupancy: dict[(r,c), building_id] lookup the core
  maintains incrementally.

  Do you accept (44×44, first-class footprints, tile-walls with is_wall flag)? 
  Or would you rather start at 22×22 to bias toward training speed in v1, 
  accepting a future port?

  Question 8 of N — RL action space

  We settled "decision-point action model" in Q4 — the agent acts only at
  decision points, not every tick. Now: what does each action look like? This
  sets the policy network's output head and is the single biggest determinant of
   training difficulty.

  A decision-point action needs to convey: (1) what to deploy, (2) where to
  deploy it, (3) when the next decision happens, plus optionally (4) spell
  casts, which are "deploy at any tile inside the base" not "at the perimeter."

  Four design options:

  1. Flat discrete. One huge categorical: every (troop_type, perimeter_cell)
  combination + (spell_type, base_cell) combination + a "wait Δt" set + an "end
  attack" action. With ~140 perimeter cells, 6 troops, 1 spell × ~1500 base
  cells, plus wait buckets, the action space is ~3000–4000 categories. Pro: one
  head, simple. Con: massive output, no spatial inductive bias, bad sample
  efficiency.
  2. Multi-discrete factored. Separate heads for (action_kind ∈ {deploy, spell, 
  wait, end}, troop_type, row, col, wait_bucket). Pro: smaller heads, each
  independently learnable. Con: agent can emit invalid combinations (deploy with
   a spell-only target tile, etc.) — needs masking.
  3. Hierarchical / two-stage. First head picks action_kind. Conditional on
  that, subsequent heads pick the relevant sub-fields (troop type if
  kind=deploy, target tile, etc.). Cleanest semantically; standard in StarCraft
  / MOBA RL papers. Pro: invalid actions impossible by construction. Con: more
  code, two-stage sampling is harder for off-policy algorithms.
  4. Spatial action via grid-shaped policy head. Output is a (C, 44, 44) tensor
  where C = num troop types + num spells + 1 (wait/end). Argmax over the whole
  tensor picks (action, row, col). Mask invalid (action, cell) pairs (e.g.,
  troops only on perimeter cells, spells only on base cells, walls block
  deploys). Pro: directly mirrors observation tensor — perfect spatial inductive
   bias for a CNN policy, and well-established for SC2-style games. Con: needs
  explicit "wait" and "end" non-spatial logits alongside; constraints expressed
  via masks.

  My recommendation: Path 4 (spatial grid policy with auxiliary scalar heads), 
  with action masking.

  Concrete shape:
  - Spatial head: (C, 44, 44) logits where C = num_troop_types + 
  num_spell_types. Each cell-channel pair is a candidate "deploy this entity at
  this cell."
  - Scalar head: small categorical over [wait_50_ticks, wait_100_ticks, 
  wait_200_ticks, end_attack] (a few wait buckets + early termination).
  - Action selection: at each decision point, agent samples either spatial or
  scalar head (gated by a top-level binary or by softmax over flattened combined
   logits).
  - Masking: an action_mask tensor of the same shape zeroes out illegal cells
  (deploys outside perimeter, deploys on already-occupied perimeter, spells with
   army-camp empty, deploys when no troops left).
  - Constraints respected: troop count limits (housing space), spell count
  limits (1 Lightning at TH6), already-deployed-this-game bookkeeping.

  Reasoning:
  - The observation will almost certainly be a (channels, 44, 44) tensor
  (multi-channel grid). Having the action be the same shape gives a CNN policy a
   one-to-one spatial alignment between "what I see at cell (r,c)" and "what I
  do at cell (r,c)." This is what made AlphaStar tractable. The agent learns
  "deploy near a defense" directly from spatial features, not by translating
  between coord systems.
  - Action masking prevents wasted exploration on illegal moves. Without masking
   the agent burns the first ~10k episodes learning that perimeter cells exist.
  - Wait buckets (50/100/200 ticks at 10 Hz = 5/10/20 seconds) are coarse enough
   that the agent doesn't have to learn millisecond timing but fine enough to
  express "drop the next wave just as my Giants tank the Cannon."

  The cost being accepted: action masking is mandatory wiring (no quick way to
  skip it), and the policy network has to handle two heads. Both are
  well-understood.

  Implementation note: since Gymnasium doesn't natively express masked-spatial
  actions cleanly, the env exposes:

  observation_space = Dict({
      "grid": Box(0, 1, (C_obs, 44, 44)),
      "scalar": Box(...),                # globals: time remaining, troops in 
  camp, etc.
      "action_mask_spatial": Box(0, 1, (C_act, 44, 44)),
      "action_mask_scalar": Box(0, 1, (n_scalar,)),
  })
  action_space = Dict({
      "spatial_action": Discrete(C_act * 44 * 44),  # flattened
      "scalar_action": Discrete(n_scalar),
      "head_select": Discrete(2),  # which head to use this step
  })

  Or more cleanly: a single Discrete(C_act*44*44 + n_scalar) flattened action —
  simpler.

  Question 9 of N — RL observation space

  The action space is a (C_act, 44, 44) spatial tensor + scalar head. The
  observation should mirror that shape so a CNN can map directly from "what's
  here" to "what to do here." The decision is which channels go in the obs grid,
   plus what scalars accompany it.

  Three flavors:

  1. Minimal raw grid. Channels = one-hot per building type + a wall channel + a
   destroyed channel. ~20 channels. No HP info, no defense state, no troop info.
   The agent has to infer everything dynamic from successive frames.
  2. Rich semantic grid (recommended baseline). Channels split into static and
  dynamic features:
    - Static (per-building type, but for the current base): one channel per
  building type (TH, Cannon, Archer Tower, Mortar, Air Defense, Wizard Tower,
  Wall, CC, storages-collectors-other) where each building's footprint cells are
   1.0. ~12–15 channels.
    - Dynamic state: building_hp_pct (HP / max HP, smeared over footprint
  cells), is_destroyed mask, defense_attacking (1 where defenses are
  mid-attack), defense_target_distance (encoded), troop_density_per_type (one
  channel per attacker troop type with troop count smeared over occupied cells),
   troop_hp_pct, projectile_in_flight mask, spell_active_radius mask. ~15–20
  dynamic channels.
    - Static map info: is_perimeter (deploy zone), is_blocked (wall/building
  occupies tile). 2 channels.
    - Total ~30–40 channels.
  3. Entity-list observation (transformer-style). A variable-length list of
  buildings + troops, each with feature vectors. Policy is a transformer or
  set-attention. Pro: scale-invariant, no fixed grid. Con: doesn't compose with
  the spatial action head you just chose, and is harder to train.

  Plus a scalar globals vector (always present alongside whichever spatial obs):
  - Time remaining (ticks normalized to [0, 1])
  - Stars earned so far
  - % destruction so far
  - Troops remaining per type (housing-space-normalized)
  - Spells remaining per type
  - Decision point counter

  Plus the action masks (already designed in Q8) provided as part of obs.

  My recommendation: Path 2 (rich semantic grid), ~30 channels, plus scalar 
  globals, plus action masks.

  Question 10 of N — Reward function

  The reward shapes what the agent actually learns. Get this wrong and the agent
   learns to game the proxy instead of the goal. Three reward design choices to
  bundle:

  1. Sparse vs shaped
    - Sparse: reward only at episode end. reward = stars * 100 + 
  destruction_pct, zero per-step. Pro: no gaming, agent optimizes the real
  objective. Con: extremely hard to learn from at v1 — most early-training
  episodes get 0 stars, no signal.
    - Shaped: per-step reward proportional to incremental progress. Pro: dense
  gradient, faster learning. Con: shaping bugs are silent and pernicious — agent
   finds exploits.
  2. What counts as "progress" in shaped reward
    - Destruction-based: Δreward = +α * (Δ destruction_pct) per tick or per
  decision point. Easy to compute, directly correlated with stars.
    - Damage-based: Δreward = +β * damage_dealt_this_step / total_base_hp.
  Smoother because partial-damage gives signal. Risk: agent learns to
  chip-damage many buildings instead of finishing key ones.
    - Building-priority weighted: different building types have different reward
   weights (TH destroyed = big bonus, Mortar destroyed = bigger than Cannon,
  storage destroyed = small). Encodes prior knowledge but requires hand-tuning.
  3. Penalties / shaping additions
    - Time penalty (-γ per tick) to encourage fast attacks
    - Troop-loss penalty (-δ * troops_lost_value) to discourage feeding troops
  to defenses
    - Stars-bonus jumps: +100 at 50% destruction, +200 at TH destroyed, +500 at
  100% destruction (the three Clash stars)

  My recommendation: a layered shaped reward, additive across components, with 
  sparse stars as the dominant signal:

  reward(t) =
      + 100  * Δ stars_earned                    # the three star thresholds:
  50% / TH-destroyed / 100%
      +   1  * Δ destruction_pct                 # smooth signal between
  thresholds
      +  20  * Δ town_hall_destroyed             # one-time +20 when TH falls
  (bumps the "TH down" star)
      -  0.001 * 1                               # tiny per-tick time penalty
      -  0.5 * troop_value_lost_this_step        # discourage feeding (value =
  housing-space cost)
  final bonus at episode end:
      + 500 if 3 stars,  + 200 if 2 stars,  + 50 if 1 star,  0 otherwise

  Reasoning:
  - Star thresholds are the real objective, weighted dominantly. No matter what
  shaping the agent finds, the 3-star end-bonus dwarfs everything.
  - Destruction-pct as smooth signal between star thresholds. Coefficient
  deliberately small (1 per percent) so it doesn't drown the star reward but
  gives gradient for early learning.
  - TH-destroyed bonus is broken out because the second star is "TH destroyed OR
   50% destruction" — both paths matter and the agent should learn either is
  acceptable.
  - Time penalty is tiny — just enough to break ties between equivalent
  strategies in favor of faster ones. Real Clash gives time-bonus stars at high
  TH levels but TH6 doesn't, so this is mild.
  - Troop-loss penalty — cautiously included. Without it, the agent learns
  "throw all troops at the wall" because total damage is what matters. With it,
  the agent learns to use Wall Breakers to open paths instead of letting Giants
  tank walls. Coefficient is small (0.5 × housing space) so it's a nudge, not a
  constraint. Risk to monitor: if too high, agent becomes risk-averse and never
  deploys; needs tuning.
  - No explicit reward for using spells — Lightning's value emerges naturally
  from the damage it deals.
  - Sparse-only would not work for v1 — you'd be training PPO from cold start on
   a hard exploration problem with terminal-only signal. That's research-grade
  hard.

  Crucially, the shaped components are all potential-based shaping (telescoping
  deltas) when expressed as Δ-quantities: the cumulative shaped reward over an
  episode equals the terminal sparse reward (by telescoping), so optimal
  policies are unchanged by shaping. This is the textbook safe-shaping pattern
  (Ng et al. 1999).

  Two values worth flagging as tunables, not gospel:
  - The troop-loss penalty coefficient (0.5) is a guess. Should be A/B'd at
  MVP-Tiny.
  - The star-bonus weights (50/200/500) are big but the relative scale matters
  more than absolute.

  Do you accept this layered shaped reward design? Two specific decisions in 
  here you might want to push back on:
  - Including the troop-loss penalty at all — some would argue let the agent

  - Building-priority weights — I deliberately didn't add per-building weights
  (e.g., extra reward for Mortar). Want them in, or trust the agent to learn
  priorities from what defenses kill its troops?

  Question 11 of N — Training distribution: where do bases come from?

  An RL agent trained on one base overfits hard. A trained agent's
  "intelligence" is essentially how many bases it has seen. Three sources of
  training bases, with very different costs and generalization properties:

  1. Hand-built canonical set. Author 20–50 TH6 bases by hand in the sandbox-web
   editor, with deliberate variety (compact vs spread, walled vs open,
  TH-centered vs TH-cornered). Pro: high-quality, label-able as "easy / medium /
   hard," directly used for human-meaningful evaluation. Con: 50 bases is not
  enough to prevent overfit. Authoring 200+ by hand is brutal.
  2. Procedural base generator. A Python tool that emits random-but-valid TH6
  bases under constraints (TH6 building cap counts, minimum spacing, walls form
  closed compartments, no overlapping footprints). Pro: infinite training data.
  Con: random ≠ realistic — generators tend to produce strategically nonsensical
   bases that teach the agent weird priors (e.g., always-open Mortars). Tuning
  the generator to make plausible bases is its own subproject.
  3. Mutation of seed bases. Start from a hand-built canonical set; apply random
   mutations (jitter buildings by ±2 tiles, swap building positions of same
  footprint, rotate/mirror, randomly add/remove walls). Pro: distributional
  bridge between hand-built and procedural — bases stay strategically plausible
  because they're rooted in human designs. Con: still bounded by the diversity
  of seed bases.

  Plus the cross-cutting decision: how many distinct bases does the MVP-Tiny
  experiment train on, and how does MVP-Real scale up?

  My recommendation: tiered approach across phases.

  - MVP-Tiny (phase 1): Train on a single hand-built base with a single troop
  type. Beat random-deploy baseline. The point is to prove the loop, not
  generalization. One base is correct here.
  - MVP-Real (phase 2): Train on a curated bank of ~30 hand-built bases plus
  mutations (rotation/mirror = free 4× / 8× expansion, jitter mutations on top).
   Hold out 5 hand-built bases for evaluation. This gives ~120–240 effective
  training bases from 30 hand-authored ones. Sufficient for first-pass
  generalization.
  - MVP-Real+ (phase 2.5, optional): If overfit is observed on the held-out set,
   build a constrained procedural generator that respects: TH6 cap counts (1 TH,
   3 cannons, 3 archer towers, 2 mortars, 2 air defenses, 1 wizard tower, 1 CC,
  75 wall segments, X storages, X collectors), wall connectivity (every interior
   compartment closed), buildable-area constraints. Use it for additional
  training distribution but always evaluate on the held-out hand-built set.
  - Cartographer phase (v2): Real-base ingestion replaces/augments synthetic
  distribution.

  The hand-built bases live in data/sample_bases/ as JSON files. The mutation
  pipeline is a small data/mutate.py that takes a base + a seed and emits a
  mutated base. The procedural generator is data/generate.py and is deferred
  until MVP-Real shows it's needed — don't build it speculatively.

  The eval protocol that this implies:

  - Every training run reports two metrics: train mean stars (across training
  bases) and eval mean stars (across the 5 held-out bases). The gap between them
   is the overfit signal.
  - The eval set is frozen — those 5 bases never enter training. Ever. Even if
  the model needs them.
  - Curriculum learning option: order bases by difficulty (easy → hard) for
  early training, then shuffle once the agent passes a threshold on the easy
  set.

  Question 12 of N — RL library, training infra, and vectorization

  Settled: Gymnasium-style env API. Now: which library actually trains, and how
  do we run it fast enough?

  Three coupled decisions:

  1. Algorithm + library
    - (a) Stable-Baselines3 (SB3) + PPO. The default. Battle-tested, supports
  masked PPO via sb3-contrib (MaskablePPO), good docs, easy callbacks. Pro: ship
   a training run on day 2. Con: SB3 internals are opinionated — custom obs Dict
   spaces work but extending policies takes effort.
    - (b) CleanRL. Single-file PPO/IMPALA/DQN implementations meant to be
  copy-edited. Pro: full control, easy to read, perfect for a research project.
  Con: you write more glue, less out-of-the-box training infra (vec envs,
  monitoring, eval callbacks).
    - (c) Custom JAX/PyTorch implementation. Pro: maximum performance and
  flexibility. Con: months of detour from the actual research question.
  2. Algorithm choice within the library
    - PPO is the safe default for this kind of structured-action problem.
  Robust, clip-protected, well-understood.
    - IMPALA / SEED RL for distributed scale — overkill for v1.
    - DQN / Rainbow — works on discrete actions but PPO with action masking is
  generally better for masked spatial actions.
    - Action-masked PPO specifically (sb3-contrib's MaskablePPO or CleanRL's
  masked variant) is required given the action mask design from Q8.
  3. Vectorization / throughput
    - The Python sim at 10 Hz, ~30 decision points per episode, full TH6 —
  assume a single env steps at maybe 100–500 episodes/sec with optimized Python
  (NumPy ops, no per-tick allocations). Need at least 10⁶–10⁸ steps total for
  PPO to converge on a hard problem.
    - Vectorized envs: run N envs in parallel. SB3's SubprocVecEnv forks N
  workers; each runs the sim independently. With 16–32 workers on a beefy CPU
  you get 16–32× throughput.
    - GPU? The sim is CPU-bound (pure Python logic). The policy network is
  GPU-friendly but tiny (CNN over 30×44×44). For v1, training on a single
  workstation with a midrange GPU + 16-core CPU is enough.

  My recommendation: SB3 + sb3-contrib's MaskablePPO + 16-worker SubprocVecEnv 
  on a single GPU box.

  Reasoning:
  - SB3 over CleanRL because this is a research project with multiple subsystems
   (sim, env, training, eval, replay viewer). You don't want to also be

  Question 15 of N — Inter-subsystem data contracts and config management

  The three subsystems exchange information through JSON files on disk (the
  spine of the architecture). Pinning their schemas down is what makes the
  subsystems independently developable. Two coupled decisions:

  1. Schema definition + validation
    - (a) Pydantic v2 models as the source of truth. Each schema is a Python
  class; JSON load/dump is automatic; validation is automatic; types are checked
   at runtime; and TypeScript types for the web side are auto-generated via
  datamodel-code-generator or by hand. Pro: single source of truth, runtime
  errors caught at the boundary. Con: TypeScript side needs a generation step or
   duplicate hand-written types.
    - (b) JSON Schema as the source of truth. Both Python (via jsonschema) and
  TypeScript (via json-schema-to-typescript) generate from the same JSON Schema
  files. Pro: language-neutral. Con: writing JSON Schema by hand is verbose;
  ergonomics worse than Pydantic.
    - (c) Plain JSON, no validation. Pro: zero overhead. Con: silent corruption,
   every data bug becomes a hours-long investigation.
  2. Config management
    - (a) Hydra. Hierarchical YAML configs, override syntax, multirun. Pro:
  standard in RL/ML research. Con: imports a lot of machinery, magic that's a
  pain to debug.

  --key.subkey=value parsing. Pro: aligns with (1a) above; one validation system
   for everything. Con: lacks Hydra's multirun ergonomics.

  Question 16 of N — Test strategy and CI

  Two coupled architectural decisions that govern how the agentic-issue workflow
   stays sane.

  1. Test pyramid for this project
    - Unit tests — small, fast, isolated. sandbox_core/grid.py, troop.py,
  combat.py, pathfinding.py are mostly pure functions; trivial to unit-test.
  Reward calculator is a pure function of state deltas; trivially testable.
  Action mask construction is testable in isolation.
    - Property tests — for the simulator core specifically. hypothesis library.
  Examples: "after any sequence of valid steps, total destruction % is
  monotonically non-decreasing," "no troop ever occupies a non-buildable tile,"
  "no two buildings overlap in footprint." These catch determinism bugs and
  invariant violations.
    - Golden replay tests — the architectural keystone for a deterministic
  simulator. Frozen (base.json, plan.json) → expected_replay.json fixtures
  committed to tests/golden/replays/. Test runs the sim and asserts the produced
   replay equals the golden one byte-for-byte (or with floating-point
  tolerance). Any unintended sim behavior change breaks the test.
    - Integration tests — full episode through the env: env.reset() → 
  loop[env.step()] → terminated. Asserts final score, replay validity, no
  exceptions. ~3–5 test cases covering different troop compositions.
    - Smoke training test — a 1000-step PPO run on the tracer base, asserts it
  doesn't crash and produces non-NaN losses. Catches integration breaks between
  env and SB3.
  2. CI / automation
    - (a) Local-only pre-commit hooks. pre-commit runs ruff + pyright + pytest
  before every commit. No GitHub Actions. Pro: fast feedback, no CI infra to
  maintain. Con: nothing enforces it on non-local agents (e.g., the ralph loop

  Question 17 (final overall-architecture question) — Cartographer's place in 
  the architecture

  Cartographer is deferred to v2, but the PRD and technical doc need to commit
  to its interface contract with the rest of the system so v1 doesn't
  accidentally make assumptions that block v2. Three decisions:

  1. Output contract
    - The Cartographer's job, ultimately, is: input = a screenshot of a Clash 
  base, output = a BaseLayout JSON. That JSON is the same Pydantic model the
  Sandbox already consumes. The Cartographer is, architecturally, just another
  producer of BaseLayout files — the simulator and RL env don't care whether a
  base came from a hand-built editor, a procedural generator, or a CV pipeline.
    - Lock this contract now: Cartographer outputs BaseLayout JSON conforming to
   the v1 schema, no special types, no separate path through the system. This is
   the single most important architectural decision for Cartographer because it
  makes v2 a pure additive feature.
  2. Pipeline shape
    - The Cartographer is a separate Python package app/cartographer/ with a CLI
   entry point: python -m cartographer ingest path/to/screenshot.png --out 
  app/data/scraped_bases/scrape_001.json.
    - High-level pipeline (each stage is a separate module, testable in
  isolation):
        - Stage 1 — Image preprocessing. Crop to play area, perspective-correct
  the isometric tilt to a top-down 44×44 grid, normalize lighting/colors.
      - Stage 2 — Tile classification. For each of the 1936 tiles, classify as
  {empty, wall, building_class, building_class_other_part}. CNN classifier
  trained on labeled tile crops.
      - Stage 3 — Building reconstruction. Group adjacent same-class tiles into
  footprints, deduplicate (a 3×3 cannon's 9 tiles all classify as "cannon" → one
   cannon entity).
      - Stage 4 — Schema emission. Write a BaseLayout JSON, validate against the
   Pydantic schema, fail loudly if any field is uncertain.
    - Each stage has a defined input/output type so v2 can iterate
  stage-by-stage.
  3. Training data for Cartographer
    - Self-bootstrapping option — use the Sandbox itself to render synthetic
  bases at the level of fidelity Cartographer expects. Generate thousands of
  (synthetic_screenshot, ground_truth_BaseLayout) pairs. Train Cartographer's
  tile classifier on those. Pro: infinite labeled data, perfect labels. Con:
  synthetic-to-real gap — depends entirely on how realistic sandbox-web's
  renderer becomes.
    - Manual labeling option — hand-label tiles on real screenshots. Pro: real
  distribution. Con: 1936 tiles × N screenshots = a lot of labels.
    - Hybrid — synthetic for pretraining, ~100 hand-labeled real screenshots for
   fine-tuning.

  My recommendation:

  - (1) Lock the output contract: Cartographer emits BaseLayout JSON, no 
  separate types. Same schema as hand-built and procedurally-generated bases.
  The downstream system is identical for all three sources. This is
  non-negotiable for clean architecture.
  - (2) Four-stage pipeline with stage-typed interfaces. Stages live as
  cartographer/preprocess.py, cartographer/classify.py,
  cartographer/reconstruct.py, cartographer/emit.py. CLI ties them together.
  Each stage is independently testable.
  - (3) Hybrid training, synthetic-first. Build the synthetic-data pipeline
  first by reusing sandbox-web's renderer to capture screenshots at
  deterministic camera angles. Pretrain on synthetic, evaluate on a small
  hand-labeled real set, fine-tune if the synthetic-to-real gap is too wide. The
   fact that we control the renderer means we can match whatever quirks of 
  synthetic data the Cartographer is trained against — a unique advantage of 
  doing both subsystems. This is also why deferring Cartographer to v2 doesn't
  cost us anything: by then sandbox-web is mature, and we can render synthetic
  data with confidence.
  - One concrete v1 commitment: the BaseLayout JSON schema is frozen at v1 (with
   schema_version field). No matter what Cartographer needs in v2, it must
  conform to this schema or migrate it. This prevents the schema from being held
   hostage by future Cartographer needs.
  - One v1 placeholder: create the app/cartographer/ package with __init__.py, a
   stub README.md saying "deferred to v2," and a one-line entry in the technical
   doc. No code. This reserves the namespace and signals architectural intent.

  The cost being accepted: zero in v1. The Cartographer architecture is
  documented but not built. This is the correct discipline — the v1 PRD
  acknowledges Cartographer exists, defines its contract, but commits no v1
  engineering effort to it.

  schema)? Or do you want to argue for any v1 Cartographer work — even just a 
  screenshot-loader prototype — to start de-risking it?
