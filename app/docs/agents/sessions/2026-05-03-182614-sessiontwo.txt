  Q5 — Grid layout, coordinate origin, deploy zone

  Three sub-decisions that all sit at the same layer.

  5a. Total grid size

  Already locked at 44×44 in technical.md. Confirming, not asking.

  5b. Coordinate origin & axis convention

  The technical doc references positions as (row, col) with sim ordering
  tie-breaks by "lowest (row, col)". I want to lock the convention precisely so
  the renderer, editor, and sim all agree.

  Recommended:
  - Tile address (r, c) with r ∈ [0, 43] = row index, c ∈ [0, 43] = column
  index.
  - Origin (0, 0) is the top-left corner of the grid (matches numpy array[r, c]
  and standard image conventions).
  - r increases downward, c increases rightward.
  - Float positions for troops: (r_f, c_f) with the same axes — a troop standing
   on the center of tile (3, 5) has float position (3.5, 5.5).
  - Building origin = the top-left tile of its footprint. A 3×3 Cannon at origin
   (10, 12) occupies (10..12, 12..14).
  - Sandbox-web does the isometric projection at render time. Sandbox-core has
  no idea the world is "isometric" — it's a square grid as far as the simulator
  is concerned.

  5c. Buildable area vs deploy zone

  This is the substantive part. In real Clash, attackers drop troops on the red
  tiles outside the village. The village is a fixed-size inner region. We need
  to pick a model:

  Option A — Static rings (recommended).
  - Inner 40×40 buildable area at rows [2, 41] × cols [2, 41].
  - Outer 2-tile ring = the deploy zone (rows 0–1 and 42–43; or cols 0–1 and
  42–43). 336 deploy tiles total.
  - Editor enforces: buildings/walls only placeable in the inner 40×40.
  - Sim/RL action mask: deploys only valid in the deploy ring.
  - Pros: trivially computable perimeter, matches real-game "village size", no
  per-base perimeter computation.

  Option B — Dynamic perimeter from village bounding box.
  - The "village" is the bounding rectangle of all placed buildings + N-tile
  buffer.
  - Deploy zone is everything outside that.
  - Pros: matches real-game behavior more closely — sparse bases have larger
  deploy zones.
  - Cons: perimeter is per-base, action mask varies, and you can degenerate to a
   tiny base in the corner with a huge deploy zone, which the agent would
  exploit.

  Option C — Per-base custom perimeter.
  - Author of each base draws/specifies the deploy zone manually.
  - Cons: extra burden on every base; mostly redundant with the bounding-box
  approach.

  My recommendation: option A. Real game is closer to B but A captures the
  strategically-relevant behavior (you must deploy from the edge), is trivial to
   mask, and matches the "44×44 with 40×40 buildable" intuition that produces
  the 44 number in the technical doc. The agent learns where to deploy along the
   edge — that's the strategic axis we care about.

  If you go with A, two sub-questions:
  - Wall placement on the boundary line. Walls on row 1 or row 42? Allowed
  (they're inside the inner 40×40)? Or do we lock walls to rows [2, 41] like
  other buildings? Real game: walls can be placed right on the village edge.
  Recommended: walls allowed on the same buildable region as other buildings 
  (rows 2–41, cols 2–41). No special boundary rules.
  - Deploy ring corners. Tiles like (0, 0) — are those deployable? Real game:
  yes, corners are deployable. Recommended: full 2-tile ring including corners 
  is deployable, total ~336 tiles.

  - Walls are full-tile blockers. A wall's footprint = its hitbox = 1×1. Troops

  - Field name: hitbox_inset: float | null (per-entity). Null → rule-derived.

  rule-derived.
  - The editor uses footprint for placement validation (no footprint overlap
  allowed), not hitbox. Two 3×3 buildings flush against each other is legal.

  Q9 — Projectile travel & attack-event timing

  How does damage propagate from "attacker decides to attack" to "target loses
  HP"? This is the rhythm of every combat event in the sim.

  9a. Projectile travel model

  Real Clash: Cannon shots are nearly instant; Archer Tower arrows are quick;
  Mortars have a famously slow parabolic arc (~3 seconds), which is a real
  strategic mechanic (players abuse the lag). Wizard Tower orbs are
  intermediate. Wall Breakers don't have projectiles (they suicide on contact).
  Lightning Spell zaps are instant.

  Three model options:

  1. Instant hit. All damage applied the tick the attack triggers. Throws away
  Mortar pre-fire mechanics. Simplest.
  2. Constant per-attacker travel time. Each attacker has a
  projectile_travel_ticks: int field. Cannon = 1 tick, Mortar = 30 ticks (3
  sec), Wizard Tower = 7 ticks, etc. Same regardless of distance. Captures the
  "Mortar is slow" mechanic at coarse fidelity.
  3. Distance-proportional travel time (recommended). Each attacker has
  projectile_speed_tiles_per_sec: float. Travel time = distance_tiles / speed * 
  10 ticks. Mortar at 3 t/s with a 6-tile lob = 20 ticks. Most faithful; matches
   real game reasonably.

  My recommendation: option 3. Add projectile_speed_tiles_per_sec: float | null
  to BuildingType (defenses) and TroopType (ranged troops). null = instant hit.
  Starter values (hand-curated since wiki probably doesn't expose):

  ┌────────────────────────┬───────────────────┬─────────────────────────────┐
  │        Attacker        │ Speed (tiles/sec) │            Notes            │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Cannon                 │ 20                │ near-instant                │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Archer Tower           │ 10                │ quick arrow                 │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Mortar                 │ 3                 │ slow arc, the strategic one │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Air Defense            │ 10                │ missile                     │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Wizard Tower           │ 6                 │ medium orb                  │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Archer (troop)         │ 10                │ arrow                       │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Wizard (troop)         │ 6                 │ orb                         │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Barbarian/Goblin/Giant │ null              │ melee, instant              │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Wall Breaker           │ null              │ melee suicide               │
  ├────────────────────────┼───────────────────┼─────────────────────────────┤
  │ Lightning Spell        │ null              │ instant zap                 │
  └────────────────────────┴───────────────────┴─────────────────────────────┘

  These are guesses. If you can eyeball the game and correct any, do; otherwise
  we ship these and tune in dedicated balance sessions.

  9b. In-flight projectile representation

  Projectiles are first-class sim entities while in flight, so the Replay can
  render them.

  Recommended: A Projectile model in WorldState:
  Projectile:
    id: int
    attacker_id: int            # building or troop id
    target_id: int              # who this is aimed at
    attacker_position: (rf, cf) # for rendering origin
    current_position: (rf, cf)  # advances per tick
    impact_position: (rf, cf)   # tile center where damage applies (target's
  hitbox center)
    damage: int                 # damage to apply on impact
    splash_radius_tiles: float  # 0 if non-splash
    splash_damages_walls: bool
    ticks_to_impact: int
    attack_kind: str            # "cannon", "mortar", "wizard_orb", etc. — for
  sprite selection

  step_tick() advances current_position by one substep along the line from
  attacker to impact, decrements ticks_to_impact, and on ticks_to_impact == 0
  resolves the hit (single-target damage or splash). The Replay's events list
  emits projectile_fired on creation and damage on impact.

  A subtle question: target dies before impact. If a Mortar fires at troop A,
  but A dies before the projectile lands, what happens?
  - Real game: The Mortar still impacts at the committed location (the location
  target was at when the shot was fired). Splash applies to whoever's there.
  - Recommended: Sim commits an impact_position at fire time and never updates
  it. On impact, splash applies in radius regardless of whether the original
  target survives. This is a real strategic mechanic — you can dodge a Mortar by
   moving troops out of the impact zone.

  For single-target projectiles (Cannon, Archer Tower), the same rule means the
  Cannon's projectile chases the fixed impact tile, not the moving troop. If the
   troop moves out, the projectile misses (does no damage, or applies 0 damage
  at impact tile with 0 splash). Recommended: Cannon projectiles also commit to 
  a tile, and a non-splash projectile that arrives at an empty impact tile does 
  no damage.

  Wait — that's actually a divergence from real Clash. In real Clash,
  single-target projectiles like Cannon arrows track the target (slight homing).
   Mortars commit to a location.

  Refined recommendation:
  - BuildingType.projectile_homing: bool (and same for ranged troops).
  - Cannon, Archer Tower, Wizard Tower, Air Defense, Archer, Wizard → homing: 
  true (projectile tracks the target until impact; if target dies, projectile
  despawns mid-air and does no damage).
  - Mortar → homing: false (commits to impact location at fire time; splash
  applies on impact).

  9c. Attack-cooldown semantics

  - An attacker has attack_cooldown_ticks — the time between consecutive attack
  triggers.
  - First attack: the moment a troop enters range and acquires a target, does it
   fire immediately, or does it wait one cooldown? Real game has a brief
  "preparation" delay.
  - Recommended: Fire immediately on acquiring a target (cooldown timer starts
  at 0 on acquisition). Subsequent attacks wait attack_cooldown_ticks. If target
   dies between trigger and impact (homing case), no damage; cooldown still
  elapses.

  Three confirmations on this question:
  1. Distance-proportional projectile travel via projectile_speed_tiles_per_sec,
   with the hand-curated starter table above. OK?
  2. Homing for single-target projectiles (Cannon/Archer/Wizard/AD);
  committed-location for Mortar (via projectile_homing: bool). OK?
  3. First attack fires immediately on target acquisition; subsequent attacks
  every attack_cooldown_ticks. OK?

  Q10 — Splash mechanics

  The PRD stub flags "splash damage falloff curve (linear, step, quadratic)" as
  an open question. I think the answer is "none" — but let me lock it precisely.

  10a. Shape

  Recommended: circular. Real Clash uses a circular splash region. Any entity
  (troop or building) whose hitbox intersects the circle takes damage.

  10b. Falloff

  Recommended: flat full damage within radius, zero outside. No linear/quadratic
   falloff in v1. Real Clash has effectively-flat splash for Mortar / Wizard /
  Wizard Tower at TH6. If we ever want to model deep mechanics like Bowler
  line-splash later, the schema accommodates it via an additional optional field
   — for v1, every splash is flat-circle.

  10c. Splash center

  The point the circle is centered on.
  - Mortar (committed-location): the impact_position chosen at fire time = the
  target's hitbox center at fire time, snapped to nearest tile center.
  - Wizard / Wizard Tower (homing): the homing projectile resolves at the
  target's hitbox center at impact tick. If target died mid-flight, projectile
  despawns (no damage, no splash). Splash center = target hitbox center on
  impact tick.
  - Wall Breaker suicide: splash center = WB's float position the tick it
  triggers.

  10d. Damage application

  - Single damage value per attack (the attacker's damage_per_shot). Same value
  applied to every entity in the radius.
  - Per-target filter on whether the entity is hit: splash_damages_walls: bool
  controls walls; troops are always damaged by attacker-source splash; buildings
   are always damaged by attacker-source splash; troops are always damaged by
  defense-source splash; buildings are never damaged by defense-source splash
  (defenses don't damage buildings).

  Implicit consequence: a Wizard Tower (defense) with splash hits troops only —
  it does not damage other buildings even if they're in its splash radius.
  Confirm this matches real game.

  10e. Distance metric

  Recommended: Euclidean distance from splash center to:
  - For a building target: the closest point on its hitbox (square inset). If
  the splash circle's edge is within splash_radius of any point of the hitbox,
  the building takes damage.
  - For a troop target: the troop's float position.

  (I'm using square hitbox here for both ground and air, since splash isn't an
  attack-against-target — it's an area effect. This is consistent regardless of
  attacker type. But: confirm. The "hitbox is square for ground, circle for air"
   rule was for targeting, which is deferred. For splash application, do you
  want the same square-vs-circle distinction, or always-square, or
  always-circle?)

  10f. Splash hits multiple troops in one cluster

  This is the whole point of splash — Mortar against a clump of Barbarians wipes
   them out.
  - Implementation: on impact, iterate every entity, check distance, apply
  damage if filter+distance match.
  - O(N) per impact at 50×50 with maybe 50 entities → trivial.

  10g. Friendly-fire model

  - A defense-source splash hits attacking troops only. It does not damage other
   defenses. (Already covered.)
  - An attacker-source splash (Wizard) hits buildings + walls in radius. Does it
   damage other troops (friendly attackers)? Real game: no — friendly fire is
  off.
  - Recommended: splash_damages_friendly_troops: false for all attacker splash.
  Hard-coded as a sim invariant since it's not a tunable.

  Three confirmations on this question:

  1. Flat circular splash, no falloff, with the splash centers as described in
  10c. OK?
  2. Wizard Tower (defense) damages only troops in its splash, never other
  buildings. OK?
  3. For splash distance to building hitboxes: always-square hitbox (regardless
  of attacker type), since splash isn't an attack-against-a-target. OK with
  always-square for splash, or do you want it to follow the same ground/air rule
   as targeting?

  Q12 — Termination & scoring

  The simulator needs unambiguous termination conditions and a precise scoring
  rule. Both feed the reward function and are testable invariants.

  12a. Termination conditions

  The episode ends when any of these become true (whichever fires first):

  1. Timer expires. Real Clash attack length is 3 minutes. At 10 Hz that's 1800
  ticks. After tick 1800, the episode is terminated (truncation in Gym terms,
  but for our purposes terminated since the score is final).
  2. 100% destruction. Every non-wall building destroyed.
  3. Agent ends voluntarily. Agent emits the end_attack scalar action. Locks in
  the current score.
  4. No more attack potential. All deployed troops dead AND
  troops_remaining_in_camps == 0 AND spells_remaining == 0 AND no in-flight
  projectiles. The attacker can't make further progress, so the sim ends. Real
  game has an "End Battle" button that the player presses; this rule is the
  auto-equivalent.

  For training: condition 4 should be checked because otherwise the env wastes
  ticks running the sim out to 1800 with nothing happening. The agent can also
  emit end_attack voluntarily to skip the wait.

  12b. Star thresholds (matches real Clash, locked)

  - stars = (1 if destruction_pct >= 50 else 0) + (1 if town_hall_destroyed else
   0) + (1 if destruction_pct >= 100 else 0)
  - Stars are computed at episode end, not continuously. The Score schema has
  stars populated only on the terminal frame.

  12c. Destruction percent calculation

  Real Clash: destruction_pct = destroyed_non_wall_buildings / 
  total_non_wall_buildings × 100. Discrete per building, not HP-weighted. Walls
  don't count.

  Recommended: Match real game exactly.
  - total_non_wall_buildings = count of BuildingPlacements where category != 
  "wall" at sim start.
  - destroyed_non_wall_buildings = count of those whose hp <= 0 at any moment.
  - TH counts as one normal building.

  A consequence: a base with 30 non-wall buildings has destruction increments of
   100/30 ≈ 3.33% per kill. The reward function uses this discrete signal — the
  "smooth Δ-destruction" in the PRD is per-building Δ, not per-tick HP.

  12d. Town-hall-destroyed flag

  - town_hall_destroyed: bool set true the tick the TH building's HP reaches 0.
  Latched — doesn't unset.

  12e. Score schema

  Confirming the Score schema fields locked in technical.md/prd.md:
  Score:
    stars: int  # 0..3, populated at terminal tick only; 0 mid-episode
    destruction_pct: float  # 0..100, continuous over episode
    ticks_elapsed: int  # current tick count
    town_hall_destroyed: bool

  12f. Edge cases

  - Base with only TH and walls. total_non_wall = 1. Destroying TH = 100%
  destruction = 3 stars. Edge case but legal.
  - Base with no Army Camps. total housing space = 0. Agent can't deploy troops.
   Sim terminates immediately on condition 4. Score = (0 stars, 0%, 0 ticks,
  false). Legal but degenerate; tracer base certainly avoids this.
  - Agent emits end_attack at tick 0 with troops still in camp. Terminates
  immediately. Legal but bad strategy. The reward function doesn't need to
  special-case this — low destruction = low reward.
  - Agent never emits any action; just emits wait until timer. Sim runs to 1800.
   Final score reflects no troops deployed. Legal.

  14a. View projection: top-down or isometric?

  Iso projection spec (lock these, then I'll move to the editor UX)

  Projection

  - 2:1 dimetric projection. Each grid tile renders as a diamond 64 px wide × 32
   px tall on screen. This is the standard "video-game isometric" — every tile
  is twice as wide as it is tall.
  - Camera direction. Grid origin (0, 0) is rendered at the top of the screen.
  Row index increases down and to the left of the screen; column index increases
   down and to the right. So the bottom of the screen corresponds to grid (43, 
  43) (or (46, 46) if you include deploy ring), and the visual "south" is +row 
  +col.

  - Visualization for a 3×3 footprint at grid origin (10, 12):
                      ▲ (top of screen, grid 0,0)
                     ╱╲
                   ╱    ╲     row+ ↓ (going down-left)
                ╱        ╲
              ◀            ▶  col+ ↓ (going down-right)
                ╲        ╱
                  ╲    ╱
                    ╲╱
                    ▼ (bottom of screen, grid max)
  - Pixel formula (top-left of tile diamond's bounding box):
    - screen_x = (col - row) * 32 + canvas_center_x
    - screen_y = (col + row) * 16

  Sprite generation specs

  - Format. PNG with alpha transparency. Single-frame static (animations are
  Phase 2+).
  - Camera angle for rendering your assets. 2:1 dimetric, light source from
  upper-left of screen, no perspective distortion (orthographic). If you're
  generating with AI tools, use the prompt anchor "video game isometric, 2:1
  dimetric projection, no perspective" and a pure-color or transparent
  background.
  - Footprint base diamond size (per N×N footprint):

  | Footprint               | Base diamond (px) | Recommended canvas (px, with
  height room) |
  |-------------------------|-------------------|-------------------------------
  ------------|
  | 1×1 (wall)              | 64 × 32           | 64 × 64
              |
  | 2×2                     | 128 × 64          | 128 × 96
              |
  | 3×3                     | 192 × 96          | 192 × 144
              |
  | 4×4 (TH, CC, Army Camp) | 256 × 128         | 256 × 192
              |

  - The "canvas with height room" gives you ~50% extra vertical pixels above the
   base so buildings can have visible structure above the diamond.
  - Sprite anchor (where the renderer pins the sprite). Bottom-center pixel of
  the canvas. Specifically: the (canvas_width / 2, canvas_height) pixel of the
  sprite image is positioned at the bottom corner of the footprint diamond on 
  screen — i.e., the south corner of the building's grid footprint.
  - Troop sprites. 48 × 64 px canvas. Anchor = bottom-center, positioned at the
  troop's float position projected to screen. Troops don't have multi-tile
  footprints — they're point entities for sim purposes, just rendered as iso
  sprites.
  - Effect sprites. Explosion / splash / bolt: 96 × 96 px transparent canvases,
  anchored center.
  - Terrain tile. 64 × 32 px grass diamond. Tiled across the buildable region.

  }

  config_hash is an SHA-256 over the canonicalized JSON of the inputs that
  determine sim output: base.json, plan.json, buildings.json, troops.json,
  spells.json, manual_overrides.json. Two replays with the same config_hash must
   be bit-identical. This is the foundation of golden-replay tests and
  determinism regression.

  19b. Event types (full list)

  deploy            - troop deployed at tile
  spell_cast        - spell cast at tile (one event per cast, not per bolt)
  bolt_struck       - one Lightning bolt landed (one event per bolt)
  attack_start      - troop or defense begins attacking a target
  projectile_fired  - projectile spawned mid-air (renderer draws it)
  damage            - damage applied to an entity (target_id, amount, source_id)
  destroyed         - entity HP reached 0 (target_id, position, was_th)
  target_acquired   - troop/defense locked onto a new target (deferred —
  pathing/targeting phase)
  target_lost       - previous target died, troop seeking new one (deferred)
  attack_end        - attacker stops attacking (target dead, out of range, etc.)
   — deferred
  end_attack        - the agent emitted end_attack
  sim_terminated    - sim finished naturally (timer / 100% destruction /
  nothing-left)

  The deferred events get added in the targeting phase. v1 of Phase 0/1 includes
   everything except those four.

  Each event has {type, tick, payload: {...type-specific fields...}}.

  19c. Float precision in serialized replays

  Troop positions are floats. Default Python serialization has full precision,
  but replays don't need it.

  serialization (precision = 0.001 tile = 0.625 px at 32 px/tile, well below
  visible). Reduces file size and makes diffs reviewable.

  config_hash is computed over the rounded values, so the hash is stable.

  19d. File size estimate

  Q20 — Determinism, performance, schema versioning (wrap-up)

  A handful of cross-cutting lock-ins that the PRD needs to commit to.

  20a. Tie-break canonical ordering

  The technical doc says ties resolve by "lowest building id, then lowest (row, 
  col)." Locking the precise rule for every tie-prone operation:

  Operation: Defense picks among multiple in-range troops
  Tie-break: Lowest troop.id
  ────────────────────────────────────────
  Operation: Splash impacts ordered for damage application
  Tie-break: Iterate by (target.id) ascending
  ────────────────────────────────────────
  Operation: Deployment order on the same tick
  Tie-break: Order in DeploymentPlan.actions[]; for env-injected actions, FIFO
    insertion order
  ────────────────────────────────────────
  Operation: Multiple buildings of the same type, multiple destroyed in same
  tick
  Tie-break: All destruction events emitted, ordered by building.id ascending
  ────────────────────────────────────────
  Operation: Destruction-percent recomputation
  Tie-break: Pure function of current WorldState; no order sensitivity
  ────────────────────────────────────────
  Operation: Building/troop ID assignment
  Tie-break: Monotonic counter, assigned at sim-start (buildings) or deploy-time

    (troops). Buildings: order in BaseLayout.placements[] → IDs 0..N-1. Troops:
    order of schedule_deployment() calls → IDs starting at N.

  Property test: "running the same (base, plan) twice produces bit-identical
  replays" lives in tests/integration/test_replay_determinism.py.

  20b. Performance budgets (locked, with the 50×50 correction)

  - Sandbox-core single-core throughput, MVP-Tiny attack: ≥100 episodes/sec.
  - Sandbox-core single-core throughput, full TH6 attack: ≥50 episodes/sec.
  (Already in technical.md; unchanged by 50×50 vs 44×44 since it's the same N
  tiles in practice.)
  - Vec env: 16 workers × 50 ep/sec ≈ 800 ep/sec aggregate.
  - Replay file size: ≤ 3 MB pretty / ≤ 1.5 MB minified per attack.
  - Replay in-memory during sim: ≤ 50 MB peak per worker.
  - Web frontend: 60fps interpolated playback.
  - Sim startup time (loader + content merge + base validation): ≤ 100 ms.

  If profiling shows the inner loop bottleneck blocks training throughput, the
  escalation order is: (1) pure-Python micro-optimizations (avoid dict lookups
  in tick loop, pre-resolve stats), (2) numba.jit on step_tick() inner pieces,
  (3) numpy-vectorize the splash/distance checks, (4) Cython or Rust port (Phase
   2.5+ only if (1)-(3) prove insufficient).

  20c. Schema versioning posture

  - Every persisted JSON has schema_version: int.
  - schema_version: 1 is the frozen v1 contract — the schemas committed at end
  of Phase 1.
  - Migrations live in app/sandbox_core/schemas.py as functions named
  migrate_<schema>_vN_to_vM(old: dict) -> dict. The reader applies migrations
  transparently when it loads a JSON whose schema_version < CURRENT.
  - A schema additive change (new optional field, new optional enum value) does
  not bump the version.
  - A schema breaking change (removed field, renamed field, semantic change)
  must bump the version and ship a migration.
  - The migrations directory has tests: for each schema, a fixture
  tests/golden/migrations/<schema>_v1.json exists; CI loads it through the
  migration chain and asserts it validates against the latest schema.

  This is committed for v1's lifetime. Phase 4 (TH7+) may or may not require a
  v2 — if it does, the migration path is well-defined; if not, additive changes
  carry forward.

  20d. Sim API edge cases

  - Sim.step_tick() after is_terminal() == True → raises SimTerminatedError.
  - Sim.schedule_deployment() after is_terminal() == True → raises
  SimTerminatedError.
  - Sim.schedule_deployment() with an action that fails validation (housing
  exceeded, etc.) → raises InvalidDeploymentError with details. Pre-validated
  plans never see this in practice.
  - Sim.advance_to(target_tick) where target_tick < current_tick → raises
  ValueError.
  - Loading a Replay that fails Pydantic validation → raises
  ReplayValidationError.
  - All these errors are caught in CLI and reported as exit-code-1 + clean
  stderr message. The RL env never lets them propagate unhandled.

  20e. Pre-deployment / attack-start

  Recommended: No pre-deployment phase. Sim starts at tick 0 with all buildings
  active, no troops, agent's first decision point is tick 0. Real-game's
  "starting" countdown doesn't exist for our v1 since we have no heroes / CC
  troops to pre-place.

  20f. Sim version vs schema version

  - schema_version = on-disk JSON contract. Frozen at v1.
  - sim_version = pyproject.toml package version. Bumped on any change to
  simulator behavior (combat math, splash radius semantics, etc.). Recorded in
  Replay.metadata.sim_version. A golden-replay test must specify the sim_version
   it was generated against; running a different version's sim against the same
  (base, plan) may produce a different replay — that's expected, and the test
  framework either updates the golden or fails loudly depending on the
  --update-golden flag.

      129 +### 5.1 Grid geometry & coordinates                           
      130 +- **Total grid: 50×50.** This is the simulator's full play are
          +a, including the deploy zone.                                 
      131 +- **Inner buildable region: 44×44.** Rows 3–46, cols 3–46. Bui
          +ldings and walls only placeable here.                         
      132 +- **Deploy ring: 3 tiles wide on each side.** Rows 0–2, rows 4
          +7–49, cols 0–2, cols 47–49. Troops only deployable here. Spell
          +s only castable in the inner buildable region (not the deploy 
          +ring).                                                        
      133 +- **Coordinate convention.** `(row, col)` with origin `(0, 0)`
          + at the **top-left**; `row` increases downward, `col` increase
          +s rightward (matches numpy `array[r, c]` and standard image co
          +nventions). Float positions for troops use the same axes.     
      134 +- **Tile centers** at half-integer coordinates: a tile `(r, c)
          +` has center at `(r + 0.5, c + 0.5)` in float coords.         
      135 +- **Building origin** = the **top-left tile of its footprint**
          +. A 3×3 Cannon at origin `(10, 12)` occupies tiles `(10..12, 1
          +2..14)`.                                                      
      136 +- **All TH6 footprints are square** (1×1 walls, 2×2 (none at T

          +H + CC + Army Camp). Schema retains `[h, w]` for forward compa
          +tibility.                                                     
      137 +                                                              
      138 +### 5.2 Building placement rules

          +les; the simulator may trust a validated layout.              
      140 +

          +are any tile. Strict.                                         
      142 +- **No mandatory spacing.** Buildings and walls may sit flush 
          +against each other.

          +n `r ∈ [3, 46], c ∈ [3, 46]`.

      147 +- **Decorations / obstacles / gem boxes / trees.** Out of scop
          +e; not modelled.                                              
      148 +- **CC contents.** `cc_contents` field is always empty in v1; 
          +validator rejects non-empty.                                  
      149 +                                                              
      150 +### 5.3 Walls                                                 
      151 +- **Storage.** Each wall is a separate `BuildingPlacement` ent
          +ry with `building_type: "wall"` and a 1×1 footprint at its ori
          +gin tile. No compressed-segment representation.               
      152 +- **Cap.** 75 walls at TH6 (from `th_caps.json`).             
      153 +- **Editor UX.** Drag-paint mode (`W` shortcut): the user hold

          +onal line. Non-axial drags resolve to L-shapes. Painting over

          +oves any building or wall.

      161 +                                                              
      162 +### 5.4 Hitbox model                                          
      163 +The hitbox is the geometric region used for *targeting* (i.e.,
          + the region a troop must be in range of to attack the building
          +). It is smaller than the footprint — a small "moat" of footpr
          +int tiles is walkable and unattackable around each non-wall bu
          +ilding.                                                       
      164 +                                                              
      165 +- **Schema field.** `BuildingType.hitbox_inset: float | null` 
          +(per entity). `null` resolves to the default rule at load time
          +; the in-memory `BuildingType` always carries an explicit floa
          +t so combat code never branches on entity name.               
      166 +- **Default rule.** `hitbox_inset = max(footprint_size / 2 − 0
          +.5, 0.5)`.                                                    
      167 +  - 1×1 (wall): inset 0.5 (the full tile is the hitbox).      
      168 +  - 2×2: inset 0.5.                                           
      169 +  - 3×3: inset 1.0.                                           
      170 +  - 4×4: inset 1.5.                                           
      171 +- **Override.** Army Camp (4×4) overrides to inset 1.0.       
      172 +- **Hitbox shape is attacker-dependent.** For ground attackers
          +, the hitbox is a **square** (footprint shrunk by `hitbox_inse
          +t` per side). For air attackers, the hitbox is a **circle** (r
          +adius = `hitbox_inset`, centered on footprint center). The "wh
          +y" is part of the deferred targeting design.                  
      173 +- **Walls.** A wall's footprint = its hitbox = 1×1 (a wall fil
          +ls the full tile). Troops cannot squeeze through a wall-and-bu
          +ilding corner.                                                
      174 +- **Implication.** Two adjacent non-wall buildings have hitbox
          +es that don't touch — a 1-tile-wide walkable corridor exists b
          +etween their hitbox edges. Pathfinding consumes this (deferred
          +); building placement must not assume hitbox-adjacency means "
          +no walk-through".                                             
      175 +

          +cation (how much, given a target?) is locked here.            
      178 +

          +n_castle`, `defense`, `wall`, `resource_collector`, `resource_
          +storage`, `army`, `builder_hut`. Drives multiplier lookup, pal
          +ette grouping, and (downstream) per-building reward weights.  
      180 +- **Multiplier schema on TroopType:** `damage_multipliers: dic
          +t[category, float]` (default `{}`) plus `damage_multiplier_def
          +ault: float = 1.0`.                                           
      181 +- **Damage formula:** `damage_dealt = troop.base_damage_at_lev
          +el × troop.damage_multipliers.get(target.category, troop.damag
          +e_multiplier_default)`.                                       
      182 +- **Wall Breaker:** `base_damage` set to the wiki's wall-damag
          +e value; `damage_multipliers: {"wall": 1.0}`; `damage_multipli

          +-game values during scraper review).                          
      183 +- **Goblin:** `damage_multipliers: {"resource_collector": 2.0,
          + "resource_storage": 2.0}`; `damage_multiplier_default: 1.0`. 
      184 +- **All other TH6 troops:** empty multipliers dict, default 1.
          +0.                                                            
      185 +- **Defenses do not have multipliers.** They apply flat `damag
          +e_per_shot` to troops.                                        
      186 +- **No friendly-troop damage** from attacker splash (Wizard's 
          +splash does not hit friendly troops). Hard sim invariant, not 
          +a tunable.                                                    
      187 +                                                              
      188 +### 5.6 Projectile mechanics                                  
      189 +Damage propagation between attack-trigger and target HP change
          +.                                                             
      190 +                                                              
      191 +- **Model.** Distance-proportional travel time. Each ranged at
          +tacker has `projectile_speed_tiles_per_sec: float | null`. Tra
          +vel time = `distance_tiles / speed × 10` ticks. `null` means i
          +nstant hit (melee, suicide, Lightning).                       
      192 +- **Starter values** (hand-curated; refined during balance ses
          +sions):                                                       
      193 +  - Cannon ≈ 20 tiles/sec (near-instant).                     
      194 +  - Archer Tower ≈ 10.                                        
      195 +  - Mortar ≈ 3 (slow arc — strategic mechanic).               
      196 +  - Air Defense ≈ 10.                                         
      197 +  - Wizard Tower ≈ 6.                                         
      198 +  - Archer (troop) ≈ 10. Wizard (troop) ≈ 6.                  
      199 +  - Barbarian / Goblin / Giant / Wall Breaker → `null`.       
      200 +- **Homing flag.** `BuildingType.projectile_homing: bool` and 
          +`TroopType.projectile_homing: bool`.                          
      201 +  - Single-target ranged attackers (Cannon, Archer Tower, Wiza
          +rd Tower, Air Defense, Archer, Wizard): `homing: true`. The pr
          +ojectile tracks the target until impact. If the target dies mi
          +d-flight, the projectile despawns and does no damage.         
      202 +  - Mortar: `homing: false`. The Mortar commits to an `impact_
          +position` at fire time (target's hitbox center snapped to near
          +est tile center). On impact, splash applies in radius regardle
          +ss of whether the original target survives. This is the real-C
          +lash "dodge a Mortar" mechanic.                               
      203 +- **In-flight projectile representation.** A `Projectile` mode
          +l in `WorldState` tracks `attacker_id`, `target_id`, `attacker
          +_position`, `current_position`, `impact_position`, `damage`, `
          +splash_radius_tiles`, `splash_damages_walls`, `ticks_to_impact
          +`, `attack_kind`. Each tick advances `current_position` along 
          +the attack→impact line. On `ticks_to_impact == 0`, the hit res
          +olves and a `damage` event emits.                             
      204 +- **Attack-cooldown semantics.** When an attacker acquires a t
          +arget, it fires immediately (cooldown timer starts at zero). S
          +ubsequent attacks elapse `attack_cooldown_ticks`. If the targe
          +t dies between trigger and impact (homing case), no damage is 
          +applied; cooldown still elapses normally.                     
      205 +                                                              
      206 +### 5.7 Splash mechanics                                      
      207 +- **Shape: circular.** Any entity whose hitbox intersects the 
          +splash circle takes damage.                                   
      208 +- **Falloff: none.** Flat full damage within radius, zero outs
          +ide. No linear/quadratic falloff in v1.                       
      209 +- **Splash center:**                                          
      210 +  - Mortar (non-homing): `impact_position` committed at fire t
          +ime.                                                          
      211 +  - Wizard / Wizard Tower / other homing splashers: target's h
          +itbox center on impact tick.                                  
      212 +  - Wall Breaker suicide: WB's float position the tick the sui
          +cide triggers.                                                
      213 +- **Distance metric for splash:** Euclidean from splash center
          + to (a) the closest point on each building's **square hitbox**
          + (always-square for splash distance — independent of attacker 
          +type), or (b) each troop's float position. If distance ≤ `spla
          +sh_radius`, damage applies (subject to filter).               
      214 +- **Friendly-fire model.** Defense-source splash hits attackin
          +g troops only (no other defenses). Attacker-source splash (Wiz
          +ard) hits buildings + walls (filter-permitting) + does NOT hit
          + friendly troops.                                             
      215 +- **Wizard Tower (defense) splash damages only troops.** Never
          + damages other buildings or walls.                            
      216 +                                                              
      217 +### 5.8 Lightning Spell                                       
      218 +- **Cast mechanics.** Agent emits `kind: "spell", entity_type:
          + "lightning", position: (r, c)` at decision-point tick `T`. On
          + tick `T`, a `SpellCast` entity is created in `WorldState` wit
          +h `cast_tick=T`, `center=(r, c)` (tile center), `bolts_remaini
          +ng=N`, `next_bolt_tick=T+1`. Each bolt fires per `hit_interval
          +_ticks`, applies `damage_per_hit` to every entity in radius, e
          +mits a `bolt_struck` event, and decrements `bolts_remaining`. 
          +When zero, the SpellCast despawns.                            
      219 +- **Per-level stats** in `spells.json`:                       
      220 +  - `radius_tiles: 2.0` (approximate; scraper authoritative). 
      221 +  - `hit_interval_ticks: 1`.                                  
      222 +  - `num_hits: 11` (constant across levels at TH6 max).       
      223 +  - `damages_walls: false`.                                   
      224 +  - `target_filter: "all_except_walls"` (ground troops, air tr
          +oops, all buildings except walls).                            
      225 +  - `housing_space: 1`.                                       
      226 +  - `levels[]`: per-level damage_per_hit values (level 1, 2, 3
          + at TH6-cap).                                                 
      227 +- **Friendly damage.** Lightning DOES damage friendly troops i
          +n radius (matches real game).                                 
      228 +- **Spell capacity at TH6.** `th_caps.json` declares `spell_ca
          +pacity_total: 2`. The action mask zeroes out spell casts when 
          +remaining capacity is 0.                                      
      229 +- **Cast position legality.** Inner buildable region only (row
          +s 3–46, cols 3–46); not in deploy ring.                       
      230 +                                                              
      231 +### 5.9 Termination & scoring                                 
      232 +- **Termination conditions** (any-fires):                     
      233 +  1. Tick ≥ 1800 (3-minute timer at 10 Hz).                   
      234 +  2. 100% destruction (every non-wall building destroyed).    
      235 +  3. Agent emits `end_attack` scalar action.                  
      236 +  4. No further attack potential: all deployed troops dead AND
          + troops_remaining_in_camps == 0 AND spells_remaining == 0 AND 
          +no in-flight projectiles.                                     
      237 +- **Star thresholds** (matches real Clash, computed at termina
          +l tick only):                                                 
      238 +  - 1 star if `destruction_pct ≥ 50`.                         
      239 +  - +1 star if `town_hall_destroyed`.                         
      240 +  - +1 star if `destruction_pct ≥ 100`.                       
      241 +- **Destruction percent.** Discrete per-building, walls exclud
          +ed:                                                           
      242 +  `destruction_pct = (count of destroyed non-wall buildings / 
          +total non-wall buildings) × 100`.                             
      243 +  TH counts as one ordinary building. A building is destroyed 
          +when its HP ≤ 0 (latched). Partial HP loss does not count.    
      244 +- **Town-hall-destroyed flag.** Set true the tick the TH build
          +ing's HP reaches 0; latched.                                  
      245 +- **`Score` schema** (computed each tick; only `stars` is non-
          +zero on the terminal tick): `stars: int (0..3)`, `destruction_
          +pct: float (0..100)`, `ticks_elapsed: int`, `town_hall_destroy
          +ed: bool`.

      276 +                                                              
      277 +### 6.5 Float precision in serialized replays                 
      278 +All `(rf, cf)` floats round to **3 decimal places** before ser
          +ialization. `config_hash` is computed over the **rounded** val
          +ues; the hash is therefore stable across float-noise.         
      279 +

      281 +- **Default mode: minified** (no whitespace, single line). Use
          +d by the sim, viewer, and RL env.                             
      282 +- **Pretty mode** with stable key ordering, used for committed
          + golden-replay fixtures so `git diff` is reviewable.          
      283 +- The reader handles both transparently.                      
      284 +                                                              
      285 +### 6.7 `config_hash`                                         
      286 +SHA-256 over the canonicalized JSON of the inputs that determi
          +ne sim output: `base.json` + `plan.json` + `buildings.json` + 
          +`troops.json` + `spells.json` + `manual_overrides.json`. Two r
          +eplays with identical `config_hash` *must* be bit-identical. T
          +he hash is the foundation of golden-replay tests and determini
          +sm regression detection.                                      
      287 +                                                              
      288 +### 6.8 `sim_version` vs `schema_version`                     
      289 +- `schema_version` — on-disk JSON contract. Frozen at v1.     
      290 +- `sim_version` — `pyproject.toml` package version. Bumped on 
          +any change to simulator behavior. Recorded in `Replay.metadata
          +.sim_version`. Golden-replay tests record the `sim_version` at
          + fixture-creation time; running a different `sim_version` agai
          +nst the same `(base, plan)` may produce a different replay — t
          +he test framework either updates the golden via `--update-gold
          +en` or fails loudly.

      297 +Public API per `technical.md` §5.1, refined here:             
      298 +

      301 +- `Sim.step_tick() -> tuple[WorldState, list[Event]]` — advanc

      308 +The `Sim` is reusable across episodes within one process via `

      309 +                                                              
      310 +### 7.2 Per-tick update order                                 
      311 +Locked per `technical.md` §3 plus refinements from this PRD:  
      312 +                                                              
      313 +1. Resolve in-flight projectiles → impact events; if homing ta
          +rget died, despawn projectile silently.                       
      314 +2. Apply scheduled deployments queued for this tick (including
          + spell casts, which spawn `SpellCast` entities).              
      315 +3. Tick active spell casts (Lightning bolts).                 
      316 +4. Update troop targeting (re-pick on target death). **[Deferr
          +ed — Phase 0 stub]**                                          
      317 +5. Advance troop positions one substep along their cached path
          +. **[Deferred — Phase 0 stub]**                               
      318 +6. Troops attack in range with cooldown ready; create projecti
          +les (or apply melee damage instantly).                        
      319 +7. Update defense targeting. **[Deferred — Phase 0 stub]**    
      320 +8. Defenses attack in range with cooldown ready; create projec
          +tiles.                                                        
      321 +9. Apply accumulated damage events; mark destroyed; emit `dest
          +royed` events.                                                
      322 +10. Update score (stars, % destruction).                      
      323 +11. Emit `sim_terminated` if termination condition fires.     
      324 +12. Increment `tick`.                                         
      325 +                                                              
      326 +### 7.3 Module boundaries                                     
      327 +- **`sim.py`** — owns `Sim`, the per-tick update orchestration
          +, the world state. Thin glue.                                 
      328 +- **`combat.py`** — pure functional damage application: `apply
          +_damage(target, base_damage, attacker, multipliers) → damage_e
          +vent`. No state.                                              
      329 +- **`splash.py`** — pure functional splash resolution: `resolv
          +e_splash(world, center, radius, damage, filter, source_kind) →
          + list[damage_event]`. No state.                               
      330 +- **`scoring.py`** — pure functional scoring: `compute_score(w
          +orld) → Score` and `is_terminal(world) → bool`. No state.     
      331 +- **`grid.py`** — tile + footprint helpers, perimeter/buildabl
          +e region masks, hitbox geometry. Square inset and circular hit
          +box helpers both live here.                                   
      332 +- **`pathfinding.py`** — **[Deferred]**. Stubs raise `NotImple
          +mentedError("path: pathfinding deferred — see app/docs/sandbox
          +/")`.                                                         
      333 +- **`projectile.py`, `spell.py`, `building.py`, `troop.py`** —
          + runtime entity types; mostly data carriers with minimal behav
          +ior on them (behavior lives in pure-functional helpers).      
      334 +- **`replay.py`** — writer (incl. float rounding, hash, minifi
          +ed/pretty mode) and reader (incl. schema migration).          
      335 +- **`content.py`** — loads the four data JSONs + `manual_overr
          +ides.json` at startup; merges; exposes immutable singletons.  
      336 +- **`schemas.py`** — every Pydantic v2 model.                 
      337 +- **`cli.py`** — `run`, `validate`, `validate-plan`, `inspect-
          +replay` subcommands.                                          
      338 +- **`tools/scrape_wiki.py`** — separate dev-time tool, never i

      339 +                                                              
      340 +### 7.4 Tie-break canonical ordering                          
      341 +| Operation | Tie-break |                                     
      342 +|---|---|                                                     
      343 +| Defense picks among multiple in-range troops (deferred mecha
          +nic, but ordering rule locked) | Lowest `troop.id` |          
      344 +| Splash damage application order | Iterate by `target.id` asc
          +ending |                                                      
      345 +| Deployment order on the same tick | `DeploymentPlan.actions[
          +]` order; for env-injected actions, FIFO insertion order |    
      346 +| Multiple buildings destroyed in same tick | Events ordered b
          +y `building.id` ascending |                                   
      347 +| Building ID assignment | Order in `BaseLayout.placements[]` 
          +→ IDs `0..N-1` |                                              
      348 +| Troop ID assignment | Monotonic counter starting at `len(bui
          +ldings)`, incremented per `schedule_deployment` |             
      349 +
